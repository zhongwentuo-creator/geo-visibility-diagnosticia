## 一、从 V1.0 到 V2.0：产品升级主线

### 1.1 V1.0 的核心能力（回顾）

V1.0 是一款**固定流水线的诊断工具**：
- 用户输入品牌 → 系统按预设顺序跑 9 个阶段 → 输出报告
- 每个阶段是独立的 Python 函数，顺序硬编码在 `main.py` 中
- 一个阶段失败，用默认值降级，不会自动调整策略
- 搜索数据是 LLM 模拟的，不是真实搜索 API
- 只能在本地运行，命令行输入

**V1.0 的局限**：
- 工具是死的：API 调用是硬编码的，不能灵活替换或扩展
- 流程是固定的：无论品牌什么情况，都跑同样的 9 个阶段
- 数据是模拟的：没有真实搜索数据支撑
- 部署是本地的：没有 Web 服务，无法分享

### 1.2 V2.0 的升级方向

V2.0 将进化为**自主决策的 GEO 诊断 Agent**：
- 不是"按固定顺序跑 9 个阶段"，而是"AI 根据当前情况，自己决定下一步做什么"
- 不是"硬编码 API 调用"，而是"AI 通过 MCP 协议，自主调用外部工具"
- 不是"本地命令行"，而是"Web 服务 + API 接口，随时可调用"
- 不是"零测试"，而是"自动化测试覆盖，确保稳定性"

**核心升级关键词**：
- **MCP**（Model Context Protocol）：AI 调用外部工具的标准协议
- **LangGraph**：让 AI 自主决策的编排框架（条件边 + 循环）
- **自主 Agent**：从"工具"到"Agent"的质变

### 第二章：产品定位升级（V2.0 是什么）

**目标回答**：V2.0 和 V1.0 有什么区别？用户获得了什么新能力？

**内容要点**：
- V2.0 一句话定位：从"诊断工具"到"自主诊断 Agent"
- 核心价值升级：从"一次性诊断"到"持续优化伙伴"
- 用户旅程变化：
  - V1.0：输入 → 等待 → 看报告（单次）
  - V2.0：输入 → AI 自主决策诊断深度 → 输出报告 + 持续追踪建议（持续）
- 新场景解锁：
  - 场景：AI 发现某品牌基建得分极低 → 自动进入"深度基建分析"分支
  - 场景：AI 发现竞品有突发事件 → 自动追加舆情追踪节点
  - 场景：用户说"预算有限，只改最关键的一点" → AI 自动精简流程，只输出最高优先级建议

### 第四章：核心能力升级（V2.0 能做什么）

#### 能力 1：MCP 工具调用（外部能力扩展）

**What**：将现有的 API 调用（Kimi、豆包、搜索）封装为 MCP 标准工具，让 AI 能自主决定调用什么工具。

**V1.0 现状**：
```python
# 硬编码：main.py 直接调用
doubao_search("听力熊 怎么样")
```

**V2.0 目标**：
```python
# MCP：AI 自主决定调用哪个工具
llm_chat_with_tools(
    "诊断听力熊的 GEO 可见度",
    tools=["search_api", "doubao_ai", "kimi_llm", "sentiment_scraper"]
)
# AI 自己决定：先搜索 → 再分析 → 再搜索竞品
```

**关键变化**：
- 工具注册化：每个外部能力是一个可插拔的 MCP Tool
- 工具可扩展：新增工具不需要改 `main.py`，只需注册新 MCP Server
- AI 自主决策：LLM 根据上下文决定调用哪个工具、传什么参数

#### 能力 2：LangGraph 自主编排（流程动态化）

**What**：从固定流水线（Stage 1→2→3→...）升级为动态决策图（Node + Edge + 条件分支）。

**V1.0 现状**：
```
Stage 1 → Stage 2 → Stage 3 → ... → Stage 9（固定顺序）
```

**V2.0 目标**：
```
开始诊断
  ↓
用户画像构建（Node）
  ↓
条件边：如果基建得分 < 40？
  ├─ 是 → 深度基建分析（额外 Node）
  └─ 否 → 正常竞品分析（Node）
  ↓
条件边：如果竞品平均得分 > 80？
  ├─ 是 → 深度竞品对标（额外 Node）
  └─ 否 → 标准 AI 搜索测试（Node）
  ↓
循环边：如果 AI 搜索无结果？
  → 换 Query 重新搜索（循环回 Stage 4）
  ↓
...动态路径...
  ↓
输出报告
```

**关键变化**：
- 条件边：根据中间数据动态决定下一步
- 循环边：某阶段失败时自动重试或换策略
- 并行边：无依赖的节点仍然并行执行
- 状态持久化：诊断过程可以暂停、恢复、检查点

#### 能力 3：真实搜索 API 接入（数据真实化）

**What**：接入 SerpAPI / Bing Search API，替代 LLM 模拟搜索。

**V1.0 现状**：
- AI 搜索测试：LLM 模拟"如果搜索这个问题，大概会得到什么结果"
- 舆情扫描：LLM 模拟"如果抓取黑猫投诉，大概会有什么内容"
- **数据不真实**： simulated ≠ real

**V2.0 目标**：
- AI 搜索测试：通过 MCP 调用 `search_api` Tool，获取真实搜索结果
- 舆情扫描：通过 MCP 调用 `sentiment_scraper` Tool，抓取真实平台数据
- **数据真实**：基于实际搜索和抓取数据计算 AIVO 分数

**关键变化**：
- 评分可信度大幅提升（从模拟到真实）
- 竞品识别更准确（基于真实搜索共现数据）
- 舆情分析更及时（基于真实平台数据）

#### 能力 4：FastAPI Web 服务（部署线上化）

**What**：从本地 Python 脚本升级为 Web API 服务。

**V1.0 现状**：
```bash
python main.py --brand "听力熊" --category "儿童AI"
# 只能在本地跑，不能分享给别人
```

**V2.0 目标**：
```bash
# 部署为 Web 服务后，任何用户都可以调用
POST /api/diagnose
Body: {"brand": "听力熊", "category": "儿童AI"}
# 返回：任务 ID → 异步执行 → 回调报告 URL
```

**关键变化**：
- 异步任务队列：长耗时诊断不阻塞 HTTP 请求
- 进度查询：用户可以轮询诊断进度
- 历史记录：多次诊断数据存入数据库，可追踪趋势
- 多用户支持：不同用户的数据隔离

#### 能力 5：自动化测试（质量保障）

**What**：为每个阶段编写 pytest 测试，确保 V2.0 代码质量。

**V1.0 现状**：零测试覆盖，靠手动运行验证。

**V2.0 目标**：
```bash
pytest tests/
├── test_s1_user_profile.py
├── test_s2_infra_eval.py
├── test_s3_competitor.py
├── test_s4_ai_search.py
├── test_mcp_tools.py
├── test_langgraph_flow.py
└── test_api_endpoints.py
```

**关键变化**：
- 每次代码变更自动验证（CI/CD 基础）
- 新功能开发有测试保障
- 回归测试防止老 Bug 复发

### 第五章：AIVO 评分升级（V2.0 怎么打分）

**目标回答**：V2.0 的评分和 V1.0 有什么不同？

**内容要点**：
- 评分算法基础不变：4 维度 × 25% 权重
- **数据来源升级**：从 LLM 模拟数据 → 真实搜索 API 数据
- **动态权重**：根据行业/品类，AI 可调整维度权重（如电商品牌更看重基建，SaaS 品牌更看重舆情）
- **趋势追踪**：多次诊断数据对比，输出分数变化曲线
- **预测模型增强**：基于真实数据训练的优化效果预测（更可信）

```
用户：追加
Agent：[自动进入深度舆情分析分支]

```

### 第七章：技术架构升级（V2.0 怎么实现）

#### 7.1 架构图（V2.0）

```
用户层（Web UI / API / Chat）
    ↓
FastAPI 服务层（异步任务队列）
    ↓
LangGraph 编排层（自主决策 Agent）
    ├── Node：用户画像构建
    ├── Node：基建评估
    ├── Node：竞品分析
    ├── Node：AI 搜索测试
    ├── Node：GEO 效果汇总
    ├── Node：舆情扫描
    ├── Node：综合总览
    ├── Node：AIVO 评分
    ├── Node：建议系统
    ├── 条件边：根据分数动态分支
    └── 循环边：失败时重试/换策略
    ↓
MCP 工具层（可插拔的外部能力）
    ├── MCP Server：搜索 API（SerpAPI / Bing）
    ├── MCP Server：豆包 AI 搜索
    ├── MCP Server：Kimi LLM 推理
    ├── MCP Server：舆情抓取
    └── MCP Server：官网检测
    ↓
数据层（持久化存储）
    ├── PostgreSQL：诊断记录、用户数据
    ├── Redis：任务队列、缓存
    └── ChromaDB：向量数据库（RAG）
```

#### 7.2 技术栈变化

| 组件 | V1.0 | V2.0 |
|------|------|------|
| 编排框架 | Python 函数硬编码 | **LangGraph**（条件边 + 循环） |
| 工具调用 | 直接 HTTP 调用 | **MCP 协议**（标准 Tool 注册） |
| Web 服务 | 无 | **FastAPI** + Uvicorn |
| 任务队列 | 无 | **Celery** + Redis |
| 数据库 | 无 | **PostgreSQL**（诊断记录） |
| 向量数据库 | 无 | **ChromaDB**（RAG 知识库） |
| 测试 | 无 | **pytest**（全阶段覆盖） |
| 部署 | 本地脚本 | **Docker** + Vercel / 阿里云 |
