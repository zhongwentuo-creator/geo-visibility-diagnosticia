## 1. 产品目标与边界

### 1.1 要解决的用户任务

让品牌/营销人员在一个页面中完成：

1. 用自然语言描述品牌与诊断目标；前端自动识别品牌、产品类型和可选官网；
2. 选择待检测的 AI 搜索平台；
3. 发起 GEO 可见度诊断；
4. 连续查看 9 个阶段的实时进度与摘要；
5. 查看 AIVO 评分摘要，并打开 JSON 或下载 HTML 报告。

页面不是创作社区、资产库或营销 Landing Page。首屏唯一主任务是“开始一次 GEO 诊断”。

### 1.2 本版本不做

- 不新增登录、历史持久化、资产管理、发现流、分享、语音或上传素材。
- 不改评分规则、9 阶段流程、后端接口或诊断数据含义。
- 不复用历史参考产品的品牌、文案、业务模块或视觉素材。
- 不使用渐变背景、玻璃拟态、发光装饰、emoji 图标或自动循环的装饰动画。

### 1.3 实现边界

| 层级 | 当前约定 |
|---|---|
| 页面 | `src/static/index.html`，单页工作台 |
| 样式 | `src/static/css/style.css`，原生 CSS 和语义 token |
| 交互 | `src/static/js/app.js`，原生 JavaScript |
| 图标 | Font Awesome 6.6 CDN（功能图标）+ `assets/geo-visibility-logo.png`（产品 Logo） |
| 实时数据 | `POST /api/diagnose` 创建任务，`GET /api/diagnose/{task_id}/stream` 接收 SSE |
| 报告 | `GET /api/diagnose/{task_id}/report`、`/report/html` |

`预览流程`是前端验证用的演示数据入口，不替代真实诊断服务。

## 2. 设计方向

### 2.1 体验定位

这是一个**轻量、可靠、可连续工作的诊断工作台**：

- 浅灰画布和白色工作区让信息、状态与结果成为视觉主体；
- 单一青绿色强调色只用于当前操作、运行状态和核心评分；
- 侧栏帮助定位会话和常用入口，但不抢占输入区域；
- 用户问题、AI 诊断过程和报告在同一个滚动区域连续出现，不跳转页面；
- 每一种状态都同时使用文本、图标和结构变化表达，颜色不是唯一线索。

### 2.2 视觉原则

1. **诊断优先**：首屏的最大视觉重量给欢迎语和输入器，不给运营信息或品牌口号。
2. **工作流可见**：诊断开始后的每一个阶段都必须出现“运行中”或“已完成”证据。
3. **安静但不空洞**：大面积留白、细边框和克制阴影建立层级；卡片只承载流程节点、报告或错误，不把每段文字都卡片化。
4. **一处强调**：一屏中的高饱和色只服务主操作和主状态，不把评分、标签和导航分别染成竞争色。
5. **可恢复**：真实接口不可用时，保留已输入内容，给出具体错误和“预览流程”替代路径。

### 字体与数字

- 正文：`PingFang SC` → `Hiragino Sans GB` → `Microsoft YaHei` → system sans-serif。
- 欢迎标题：`30–42px / 1.16`，字距 `-0.045em`，桌面端居中。
- 模块标题：`15–20px`；正文与按钮：`13–14px`；元信息：`11–12px`。
- 阶段耗时和评分数字可使用等宽/表格数字；中文正文不得使用等宽字体。

### 间距、圆角与阴影

- 间距基线：`4 / 8 / 12 / 16 / 20 / 24 / 30px`。
- 小按钮和输入字段：`8–9px`；阶段节点：`12px`；报告、输入器和弹窗：`16–20px`。
- 默认用 `1px` 边框区分层级；只有输入器、报告、流程节点与弹窗使用低透明双层阴影。
- 不使用统一大圆角卡片堆叠，不把阴影用作所有模块的默认边界。

## 4. 信息架构与布局

### 4.1 桌面端（≥ 901px）

页面采用三列工作台：

| 区域 | 宽度/行为 | 内容 |
|---|---|---|
| 图标栏 | `72px` 固定 | 首页标识、诊断/记录/洞察导航、帮助入口 |
| 会话栏 | `244px` 固定 | 新诊断、最近诊断示例、会话数据提示 |
| 主工作区 | 自适应 | 顶部状态、可滚动诊断区、固定在底部的输入器 |

- 主工作区顶部最小高度 `72px`，只显示位置、连接状态和诊断记录入口。
- 空状态时欢迎区距顶部约 `64–176px`；桌面与平板端输入器距工作区底部 `280px`，形成更集中的首屏任务焦点。
- 诊断开始后，欢迎区隐藏；用户问题先作为右对齐消息进入滚动区，AI 诊断回复以独立容器承载规划、流程节点与报告。
- 对话态下输入器回到主工作区底部，保留滚动区底部空间，便于用户在完成后继续发起新的诊断。
- 输入器固定在主工作区底部，滚动区必须为其预留 `240px` 以上的底部空间。
- 桌面与平板端主工作区固定为视口高度；诊断节点只在滚动区内增长和滚动，不得撑高工作区或遮挡输入器。

### 4.2 平板（641–900px）

- 保留 `64px` 图标栏，隐藏会话栏。
- 主工作区和输入器左右内边距为 `22px`。
- 不通过压缩文字或缩小点击目标来保留会话栏。

### 4.3 移动端（≤ 640px）

- 隐藏桌面会话栏；图标导航改为底部导航，高 `64px`。
- 顶部高度缩至 `56px`，连接状态可隐藏，保留诊断记录入口。
- 输入配置改为两列，官网字段独占一行；输入器圆角降为 `16px`。
- 阶段节点改为两列，耗时放在第二列下方；评分维度的标签列缩窄。
- 页面不得产生非预期横向滚动；操作目标保留可触控尺寸。

## 5. 组件规范

### 5.1 图标栏与会话栏

- 图标栏的导航项结构为“图标 + 可见文字”；默认使用黑灰线性图标，当前项使用黑色面性图标、浅背景和 `aria-current="page"`。左上角只放 GEO 产品 Logo，不使用通用图标替代。
- 会话栏的“新诊断”是唯一的会话级主入口。最近项仅用于预填品牌与产品类型，不代表真实历史数据。
- 帮助、收起等纯图标操作必须有 `aria-label`；hover 只能提高可点击感，不改变布局。

### 5.2 欢迎区与输入器

- 欢迎区包含一句任务标题和一段不超过两行的解释；不保留短标语，避免堆放能力卡、Banner 或广告式文案。
- 欢迎说明使用 `14px` 和输入框示例文字相同的浅灰色，标题保持页面唯一的高视觉权重。
- 输入器由三部分组成：诊断意图文本区、模式/平台控件、操作区。品牌、产品类型和官网字段收在“手动补充”区域，默认不要求用户逐项填写。
- 用户输入自然语言后，前端优先通过轻量意向识别提取品牌、产品类型和官网并写入任务数据；不在输入区额外展示说明或识别标签。
- “手动补充”后的问号图标在悬停或键盘聚焦时，说明自动识别逻辑与补充入口。
- 品牌名称和产品类型仍是创建任务的必填数据；若无法从描述中识别，展开手动补充区并给出具体提示。官网可选。
- 当前只有`全量诊断`一种模式；展示模式控件是为了固定工作流语义，未新增模式前不伪造可选能力。
- `开始诊断`是唯一高强调按钮，默认黑色；hover 为黑色 `80%` 透明度。运行中改为“诊断中”并禁用重复提交。
- `预览流程`为低强调按钮，仅用于展示前端阶段/报告体验，应明确标注为演示数据。
- 使用 `Ctrl/Command + Enter` 提交表单；长文本不应盖住配置字段或主按钮。

### 5.3 诊断过程

| 元素 | 视觉与交互要求 |
|---|---|
| 用户问题 | 提交后以右对齐、浅灰消息气泡回显，输入器清空并移至页面底部 |
| AI 诊断回复 | 以独立容器承载诊断规划、进度、阶段节点和报告，避免流程卡片散落在页面中 |
| 任务引导节点 | 显示品牌、请求摘要和搜索图标，作为流程起点 |
| 进度摘要 | 显示已完成阶段数、横向进度条和百分比；不只用旋转图标表示等待 |
| 阶段节点 | 固定顺序为 1–9；显示阶段名称、状态摘要、运行中/完成图标和耗时 |
| 运行中节点 | 使用浅青背景、边框与 spinner；不改变节点尺寸 |
| 已完成节点 | 使用成功色图标、完成摘要和耗时；保留此前节点，形成可审查轨迹 |
| 错误节点 | 使用危险色、明确原因和可执行替代（重试或预览流程），不得清空输入 |

诊断滚动区带 `aria-live="polite"`，且可通过键盘获得焦点；新事件自动滚到底部，但不创建全屏遮罩或打断用户输入。

### 5.4 报告摘要

- 报告仅在 `complete` 事件后出现，包含 AIVO 分数、等级、最多四个维度和报告操作。
- 总分使用青绿色大数字；维度采用同一强调色的横向条形图，不能用四种互相竞争的颜色。
- JSON 报告与 HTML 报告操作必须指向当前 `task_id`；预览流程中应提示不存在真实报告。
- “新建诊断”回到初始工作区，不保留上一轮流程节点。

### 5.5 帮助弹窗

- 使用原生 `<dialog>`；标题、关闭按钮和确认按钮均有明确语义。
- 内容只解释 GEO 和当前 9 阶段工作流，不扩展为产品介绍页。

## 6. 状态与数据契约

### 6.1 前端状态

| 状态 | 页面行为 |
|---|---|
| 初始 | 显示欢迎区、空输入器、准备就绪连接状态 |
| 意向识别 | 用户输入描述后自动提取品牌、产品类型、官网；无法识别时由用户打开手动补充 |
| 预填 | 点击最近项后填入品牌、产品类型和诊断意图，并将焦点移到输入区 |
| 提交中 | 回显用户问题，创建 AI 诊断回复容器和 0% 进度；输入器移至底部，按钮禁用并显示“正在连接实时进度” |
| SSE 运行中 | 依次追加 `stage_start` 与 `stage_complete` 节点，连接状态显示“诊断进行中” |
| 成功 | 关闭 EventSource，显示完整进度、报告摘要和报告操作 |
| 接口异常 | 关闭 EventSource，保留输入和已产生的流程节点，显示具体错误与预览替代 |
| 预览 | 使用演示数据顺序渲染 9 阶段，报告操作不指向真实文件 |
| 减少动效 | 关闭进入、脉冲、进度条过渡等装饰动效，状态信息仍完整可见 |

### 6.2 SSE 事件映射

| 后端事件 | 前端表现 |
|---|---|
| `start` | 确认已连接，更新请求品牌文案 |
| `stage_start` | 新建或更新对应阶段节点为运行中 |
| `stage_complete` | 写入摘要和耗时，更新为已完成，推进进度 |
| `complete` | 读取 AIVO 分数/等级，再请求报告数据并渲染摘要 |
| `error` | 关闭连接并呈现错误节点，不删除已有诊断证据 |

## 7. 可访问性与动效

- 使用 `aside`、`nav`、`main`、`header`、`section`、`form`、`dialog` 等语义元素。
- 所有表单字段必须有可见名称或可访问名称；placeholder 不能是唯一标签。
- 所有图标按钮提供 `aria-label`；导航当前状态有 `aria-current`。
- `:focus-visible` 使用至少 `2px` 的青绿色外轮廓，不能只靠阴影或颜色变化。
- 关键文字、错误和状态保持可读对比度；成功/错误状态同时使用文案和图标。
- 支持键盘完成导航、填写、选择平台、提交、打开/关闭帮助和重新开始。
- 动效仅用于新节点进入、进度推进和运行状态提示，时长不超过 `700ms`；`prefers-reduced-motion: reduce` 下统一降级。

## 8. 验收标准

### 首屏与输入

- Given 用户首次打开页面，when 页面加载完成，then 能在首屏看到“GEO 可见度诊断”的任务说明、自然语言输入区、平台选择和开始诊断按钮。
- Given 用户描述了品牌、品类或官网，when 停止输入，then 页面自动写入已识别信息，用户无需逐项填写字段，也不额外打断输入区。
- Given 用户未能提供或识别必填数据，when 尝试提交，then 页面展开手动补充区并说明缺失项，且不会创建诊断任务。
- Given 用户选择最近项，when 点击任一条目，then 品牌、产品类型和诊断意图被预填，输入区获得焦点。

### 实时流程

- Given 后端返回 `task_id`，when SSE 开始推送，then 用户能在同一页面看到 1–9 阶段按顺序出现，且每个节点有状态、摘要和耗时。
- Given 诊断运行中，when 用户重复点击提交，then 不会创建第二个任务。
- Given 后端发生错误，when 前端收到 `error`，then 已输入字段和已完成阶段保留，页面给出可理解的下一步。

### 报告与响应式

- Given 收到 `complete`，when 报告数据可用，then 显示总分、等级、最多四个维度和报告操作。
- Given 页面宽度低于 `640px`，when 用户填写并提交诊断，then 底部导航、输入字段和主按钮均可见且没有横向溢出。
- Given 用户使用键盘或开启减少动效，when 执行核心任务，then 焦点顺序清晰、状态可读、核心流程不依赖动画。

## 9. 维护规则

1. UI 修改应同时检查 `index.html`、`style.css`、`app.js` 是否仍与本规范一致。
2. 新增模式、状态、数据字段或页面区域前，先更新本文件中的边界、状态表和验收标准。
3. `DESIGN_1.md` 是历史参考，不能覆盖本规范中的 GEO 业务文案、布局和数据契约。
4. 设计验收以浏览器渲染结果为准；HTML/JS 静态检查不能替代视觉与交互验证。
