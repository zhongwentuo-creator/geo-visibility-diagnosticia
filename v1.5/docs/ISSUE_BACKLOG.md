# GEO 可见度诊断师 V1.5 问题与改进待办池

> 更新：2026-07-18
> 用途：集中记录 V1.5 的交互体验、业务准确性、稳定性与技术债问题，并为后续版本选择改进项。它不是当前开发排期，也不替代验收记录。

## 1. 文档定位

在软件开发中，这类清单的标准叫法是 **Issue Backlog** 或 **Improvement Backlog**，中文可称“问题与改进待办池”。`Better case` 不是常用的软件研发术语。

文档之间的边界如下：

| 文档 | 回答的问题 |
| --- | --- |
| [ISSUE_BACKLOG.md](ISSUE_BACKLOG.md) | 目前有哪些问题、风险或改进机会，为什么值得处理？ |
| [plan.md](plan.md) | 当前决定做什么、由哪个节点执行、完成状态如何？ |
| [ACCEPTANCE.md](ACCEPTANCE.md) | 这次验收实际发生了什么，是否通过？ |
| [SDD.md](SDD.md) | 当前系统是如何实现的，以及它有哪些运行边界？ |

当一个待办项被确认进入开发，才在 `plan.md` 创建对应任务；完成后，在 `ACCEPTANCE.md` 留下验证证据，并回写本文件状态。不要仅因“可能有问题”就改动代码。

## 2. 记录规则

### 2.1 类型

| 类型 | 含义 |
| --- | --- |
| `Bug / 稳定性` | 已观察到的功能不符合预期或中断。 |
| `UX / 交互体验` | 用户完成任务时存在理解、反馈、效率或可用性问题。 |
| `业务准确性` | 诊断结果、数据口径或业务承诺可能误导用户。 |
| `技术债` | 当前可运行但在扩容、维护或稳定性上存在已知边界。 |
| `待复现` | 有历史线索或合理怀疑，但尚未在 V1.5 当前版本验证；不能当作已确认 Bug。 |

### 2.2 优先级与状态

| 优先级 | 定义 |
| --- | --- |
| `P0` | 阻塞发布、核心诊断或结果读取。 |
| `P1` | 明显影响主要用户体验、业务可信度或稳定性。 |
| `P2` | 有价值但不阻塞当前版本。 |
| `P3` | 记录观察，等待更多证据或未来版本评估。 |

状态使用：`待分诊`、`待复现`、`已确认`、`已排期`、`进行中`、`待验收`、`已验证关闭`、`接受限制`、`暂不处理`。

每条记录必须包含：唯一 ID、类型、优先级、状态、现象或影响、证据来源、下一步和关闭条件。涉及 API Key、用户隐私或生产密钥时，只记录安全的证据位置，不能复制敏感值。

## 4. 发布完成后的使用方式

本次发布验收通过不自动等于“解决所有问题”：

- 已完成的桌面、手机与 24 小时复测事实见 [ACCEPTANCE.md](ACCEPTANCE.md) 与 [NODE5_EXECUTION.md](NODE5_EXECUTION.md)；
- 若新版本复测确认问题，先在本文新增 `V15-xxx`，再判断是否建立新的 `plan.md` 任务；
- 未被选中的条目不应被暗中开发，也不应反向否定已通过的 V1.5 发布验收。

后续迭代完成后，统一更新本文、`plan.md`、`ACCEPTANCE.md` 和 [SDD.md](SDD.md)：问题状态、实施内容和验收事实必须互相一致。
