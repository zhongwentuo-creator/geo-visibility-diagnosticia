## 一、版本定位：V1.5 是什么？

### 1.1 一句话定义

> **V1.5 = V1.0 的完整能力 + 对话式 Web 体验层**

V1.0 已经能跑通 9 阶段诊断、输出 AIVO 评分和 HTML 报告。
**V1.5 不做任何诊断逻辑的改变**——只给 V1.0 套一个"对话壳"，让用户能用**自然语言对话**的方式触发诊断，并在对话中实时看到进度、结果和报告。

### 1.2 V1.0 vs V1.5 对比

| 维度 | V1.0（已冻结） | V1.5（本次升级） |
|------|--------------|-----------------|
| **触发方式** | 命令行输入参数 | 网页对话输入 |
| **交互形式** | 静默运行，结束输出文件 | 对话中实时反馈进度，流式输出卡片 |
| **体验载体** | 本地终端 | 浏览器网页 |
| **报告查看** | 本地打开 HTML 文件 | 网页内嵌展示 + 下载链接 |
| **核心逻辑** | 9 阶段 Python 流水线 | **完全不变**，只加 API 包装和前端壳 |
| **技术栈新增** | 无 | FastAPI + 前端聊天框 + SSE 流式推送 |
| **部署** | 仅本地 | **可部署到 Vercel/服务器**（URL 可分享） |

### 1.3 为什么先 V1.5 而不是直接 V2.0？

| 原因 | 说明 |
|------|------|
| **小步快跑** | V1.5 只加体验层，不动核心逻辑，风险极低 |
| **快速验证** | 2-3 天就能看到"对话式诊断"效果，验证用户是否买单 |
| **积累数据** | 用户使用后收集反馈，指导 V2.0 的功能优先级 |
| **Vibe Coding 链路闭环** | 从"本地脚本"到"可部署的 Web 服务"，跑通 DAY-01 的完整链路 |
| **课程对标** | V1.5 对应 VibeCoding 课程的 DAY-01（前端 + 部署）和 DAY-02（前后端打通） |

### 1.4 V1.5 不是 V2.0

**V1.5 明确不做**：
- ❌ 不改 MCP 工具调用（仍是硬编码 API）
- ❌ 不改 LangGraph 自主编排（仍是固定 9 阶段）
- ❌ 不改搜索数据源（仍是 LLM 模拟）
- ❌ 不加 RAG 向量库
- ❌ 不优化 AIVO 算法

**V1.5 只做**：
- ✅ 给 V1.0 加 Web 对话壳
- ✅ 让用户能在浏览器里对话式诊断
- ✅ 部署上线，可分享 URL

## 二、产品定位：V1.5 是什么产品？

### 2.1 一句话价值主张

> 在浏览器里，像聊天一样完成品牌 GEO 可见度诊断——输入品牌名，AI 自动跑完全程，实时反馈进度，最后输出评分报告和优化建议。

### 2.2 核心用户场景

#### 场景：李总监的季度品牌体检

**V1.0 体验**：
1. 打开终端
2. 输入 `python v1.0/src/main.py --brand "听力熊" --category "儿童AI"`
3. 等待 60-120 秒（屏幕黑着，不知道在干嘛）
4. 终端显示 "诊断完成"
5. 手动找 output 目录里的 HTML 文件打开

**V1.5 体验**：
1. 打开网页 `https://geo-diagnosis.vercel.app`
2. 输入："帮我诊断一下听力熊的 GEO 可见度"
3. 系统回复："好的，正在分析..." + 实时进度卡片
4. 看到每个阶段的结果：用户画像 ✓、基建评估 ✓、竞品分析 ✓...
5. 最后系统输出：AIVO 总分 74/100 + 优化建议
6. 点击"下载报告"或"查看详情"

**差异**：从"技术工具"变成"产品体验"。

### 2.3 目标用户（V1.5 不变，但新增触达方式）

| 用户 | V1.0 触达 | V1.5 新增触达 |
|------|----------|-------------|
| 品牌市场负责人 | 技术团队跑脚本 | 直接打开网页使用 |
| 数字营销经理 | 技术团队跑脚本 | 直接打开网页使用 |
| 品牌运营专员 | 技术团队跑脚本 | 直接打开网页使用 |
| **中小企业主** | 无法使用（不会命令行） | **新增用户** |
| **产品经理** | 无法使用（需要 API） | **新增用户**（可以调用 API） |

| **对话记忆** | 诊断完成后用户可追问"为什么基建分低""竞品数据在哪" | 连续对话 |

## 四、技术架构：V1.5 怎么实现？

### 4.1 架构图

```
用户浏览器
    │
    │ HTTP / WebSocket / SSE
    ▼
┌─────────────────────────────────────┐
│ 前端层（对话界面）                    │
│ • React/Vue 或纯 HTML+JS            │
│ • 聊天框组件                        │
│ • 进度卡片渲染                      │
│ • SSE 客户端（接收实时推送）         │
└─────────────────────────────────────┘
    │ API 请求（POST /diagnose）
    ▼
┌─────────────────────────────────────┐
│ 后端层（FastAPI）                    │
│ • POST /diagnose — 启动诊断任务      │
│ • GET /diagnose/{task_id} — 查进度   │
│ • GET /diagnose/{task_id}/stream —  │
│   SSE 流式推送进度                   │
│ • 异步执行 V1.0 流水线（asyncio）    │
└─────────────────────────────────────┘
    │ 调用 V1.0 核心代码
    ▼
┌─────────────────────────────────────┐
│ V1.0 引擎层（完全不变）              │
│ • v1.0/src/main.py（9 阶段流水线）            │
│ • v1.0/src/stages/（9 个阶段模块）             │
│ • v1.0/src/utils/（API 封装、JSON 修复）      │
│ • v1.0/src/report/template.html（报告模板）   │
└─────────────────────────────────────┘
    │ 输出报告文件
    ▼
┌─────────────────────────────────────┐
│ 存储层（临时）                       │
│ • 输出目录 output/（自动清理）        │
│ • 或内存缓存（单用户会话）            │
└─────────────────────────────────────┘
```

### 4.2 技术选型

| 组件 | 选型 | 说明 | VibeCoding 课程对应 |
|------|------|------|-------------------|
| **后端框架** | FastAPI | 异步 Python Web 框架，自动 API 文档 | DAY-02 前后端打通 |
| **异步任务** | asyncio | V1.0 已用，直接复用 | DAY-01 编程基础 |
| **流式推送** | SSE（Server-Sent Events） | 单向推送进度，比 WebSocket 简单 | DAY-02 实时通信 |
| **前端框架** | 纯 HTML + JS（或 Vue 3） | 越简单越好，V1.5 追求速度 | DAY-01 前端基础 |
| **CSS 样式** | Tailwind CSS 或纯 CSS | 快速美化，响应式 | DAY-01 样式设计 |
| **部署** | Vercel / 阿里云 | 一键部署，公网可访问 | DAY-01 部署上线 |
| **会话管理** | 内存字典（单用户） | V1.5 不持久化，每次刷新重新开始 | 暂不需数据库 |

### 4.3 为什么不用 WebSocket 用 SSE？

| 对比 | SSE | WebSocket |
|------|-----|-----------|
| 方向 | 服务器 → 客户端（单向） | 双向 |
| 复杂度 | 简单，HTTP 协议 | 需要握手，更复杂 |
| 适用场景 | 服务器推送进度、通知 | 实时聊天、游戏 |
| V1.5 需求 | 只需要服务器推进度给用户 | 过度设计 |
| 课程对标 | DAY-01 足够 | 需要 DAY-02 深入 |

**结论**：V1.5 用 SSE 就够了，简单、够用、好维护。

## 五、API 设计

### 5.1 核心接口

#### POST /api/diagnose — 启动诊断

```json
// 请求
{
  "brand": "听力熊",
  "category": "儿童AI对话智能体",
  "website": "https://www.tinglexiong.com",
  "platform": "doubao"
}

// 响应
{
  "task_id": "GEO-20260714-123456",
  "status": "running",
  "estimated_time": 120
}
```

#### GET /api/diagnose/{task_id}/stream — 流式获取进度（SSE）

```
event: stage_start
data: {"stage": 1, "name": "用户画像", "status": "running"}

event: stage_complete
data: {"stage": 1, "name": "用户画像", "status": "success", "elapsed_ms": 1200, "summary": "识别3类用户"}

event: stage_start
data: {"stage": 2, "name": "基建评估", "status": "running"}

...（每个阶段触发）...

event: complete
data: {"task_id": "GEO-20260714-123456", "status": "success", "aivo_score": 74, "report_url": "/api/diagnose/GEO-20260714-123456/report"}
```

#### GET /api/diagnose/{task_id}/report — 获取报告

```json
// 响应：HTML 内容或 JSON 数据
{
  "html_content": "<!DOCTYPE html>...",
  "json_data": {...}
}
```

## 七、V1.5 实现路线图
