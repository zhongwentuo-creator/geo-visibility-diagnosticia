## 1. 产品概述

### 1.1 产品定义

**GEO 可见度诊断师**是一款面向品牌市场团队的 AI 智能诊断 Agent，通过模拟真实用户在 AI 搜索平台（豆包、ChatGPT、Perplexity 等）中的查询行为，系统评估品牌在生成式引擎中的信息可见度与推荐权重，输出可落地的优化策略。

### 1.2 核心概念：GEO（Generative Engine Optimization）

| 维度 | SEO（搜索引擎优化） | GEO（生成式引擎优化） |
|------|---------------------|----------------------|
| **优化目标** | 网页在搜索结果中的排名 | 品牌信息在 AI 回答中的被引用率与推荐权重 |
| **评估场景** | 关键词排名、流量、点击率 | AI 是否主动提及品牌、如何描述品牌、竞品对比中的位置 |
| **内容形态** | 网页内容、Meta 标签 | 结构化数据、权威媒体背书、用户口碑、百科卡片 |
| **竞争维度** | 与网页竞争排名 | 与品牌在 AI 训练数据中的"认知密度"竞争 |

> **Why 现在**：随着 ChatGPT、豆包、Perplexity 等 AI 搜索平台日活突破亿级，用户获取品牌信息的首选路径正从"搜索→点击网页"转变为"直接提问→获取 AI 总结答案"。品牌若未能在 AI 回答中被准确引用，等同于在下一个时代的"搜索结果首页"中消失。

### 1.3 价值主张

```
【一句话价值主张】
帮助品牌量化自己在 AI 原生时代的"数字存在感"，并提供可执行的优化路径。
```

| 用户痛点 | GEO 诊断师的解决方案 | 可量化收益 |
|----------|---------------------|-----------|
| 不知道品牌在 AI 平台中被如何描述 | 多平台实测搜索，输出品牌提及率与引用原文 | 可见度评分（0-100） |
| 不了解竞品在 AI 搜索中的优势 | 同步执行 3-5 家竞品对标测试 | 竞品对比矩阵与差距分析 |
| 缺乏 GEO 优化的系统性方法论 | 基于诊断结果输出优先级行动清单与路线图 | 明确的时间线与预期得分提升 |
| 无法追踪 GEO 优化效果 | 结构化 JSON 数据支持二次加工与持续监测 | 可对比的历史数据基线 |

## 3. 核心系统架构与 8 阶段流水线

### 3.1 系统架构总览

GEO 可见度诊断师采用**分层管道架构（Layered Pipeline Architecture）**，由输入层、诊断引擎层、数据能力层、输出层四个核心层级组成。

```
│  │   (必填)     │  │   (必填)     │  │   (可选)     │       │
│  └──────────────┘  └──────────────┘  └──────────────┘       │
└──────────────────────────┬──────────────────────────────────┘

│  │              8 阶段流水线（Stage 1 → Stage 8）          │  │
│  │  ┌─────────┐ ┌─────────┐ ┌─────────┐ ┌─────────┐    │  │

```

### 3.2 架构设计原则

| 原则 | 说明 | 具体体现 |
|------|------|----------|
| **管道化（Pipelining）** | 每个阶段产出标准化的中间数据，下游阶段只消费、不依赖上游实现 | Stage 4 的 AI 搜索数据不依赖 Stage 3 的具体竞品识别算法 |
| **并行化（Parallelization）** | 无数据依赖的阶段可并行执行，压缩总耗时 | Stage 2 基建评估与 Stage 3 竞品分析可并行启动 |
| **容错降级（Graceful Degradation）** | 单个阶段失败时，下游阶段可基于降级数据继续运行 | 舆情平台不可用时，舆情健康度维度标记为"数据缺失"而非中断流水线 |
| **可观测性（Observability）** | 每个阶段记录状态、耗时、数据量级，支持诊断和优化 | 每阶段输出包含 `stageStatus`、`elapsedMs`、`recordCount` |

### 3.3 8 阶段流水线详解

#### Stage 1：用户画像构建（USER_PROFILE）

**What**：基于品牌名称和产品类型，构建三类核心用户画像及其搜索意图矩阵。每类画像包含：用户身份标签、核心需求场景、典型搜索 Query 列表（5-8 条）。

**Why**：GEO 优化的起点不是品牌自己说什么，而是**用户在 AI 平台中问什么**。不同用户群体的搜索意图差异直接决定了后续 AI 搜索测试的 Query 设计。

**How**：
1. **语义解析**：解析产品类型，映射到行业知识图谱中的标准品类节点
2. **用户聚类**：基于品类特征，生成 3 类典型用户（如：潜在购买者、现有用户、行业观察者）
3. **Query 生成**：为每类用户生成 5-8 条自然语言搜索 Query，覆盖信息型、导航型、交易型意图
4. **交叉验证**：通过搜索平台 API 预检 Query 的语义有效性

**输出示例**：

```json
{
  "userProfiles": [
    {
      "profileId": "UP001",
      "label": "儿童家长（潜在购买者）",
      "demographics": "30-45岁，一二线城市，关注教育科技产品",
      "searchIntent": ["信息型", "比较型"],
      "queries": [
        "儿童AI学习机哪个品牌好",
        "听力熊和阿尔法蛋对比",
        "适合3岁孩子的AI对话玩具推荐"
      ]
    }
  ],
  "totalQueries": 18
}
```

---

#### Stage 2：品牌基建评估（INFRA_EVAL）

**What**：全面扫描品牌在互联网上的"数字基础设施"：官网健康度、自媒体矩阵覆盖、权威媒体报道存量。

**Why**：基建是 GEO 的根基——AI 搜索引擎优先引用结构清晰、来源权威、信息完整的网页。

**How**：

| 评估子项 | 检测内容 | 数据来源 | 评分方法 |
|----------|----------|----------|----------|
| **官网健康度** | 可访问性、HTTPS、加载速度、移动端适配、结构化数据（Schema.org）、关于/产品/联系页面完整性 | 网页抓取 + Lighthouse 指标 | 6 项检查点，每项满分，加权求和 |
| **自媒体矩阵** | 微信公众号、抖音、小红书、B站、微博等平台的官方账号存在性与活跃度 | 平台搜索 API | 账号存在得基础分，近 90 天有更新得活跃分 |
| **权威媒体** | 36氪、DoNews、极客公园、行业垂直媒体等的专业报道存量 | 媒体库检索 + 搜索引擎 | 按报道数量、媒体权威性、报道时间衰减加权 |

**基建评分计算示例**：

| 子维度 | 权重 | 听力熊得分 | 满分标杆 |
|--------|------|------------|----------|
| 官网健康度 | 40% | 62 | 100 |
| 自媒体矩阵 | 35% | 48 | 100 |
| 权威媒体 | 25% | 55 | 100 |
| **加权总分** | 100% | **55** | — |

#### Stage 3：竞品对标分析（COMPETITOR）

**What**：自动识别 3-5 家同行业核心竞品，并建立"同条件测试基准"。竞品选择遵循**可对比性原则**（同品类、同价位带、同目标用户），而非单纯市场份额。

**Why**：GEO 可见度是相对概念。品牌得分 60 分，在行业内若竞品平均 80 分，则实际处于竞争劣势。

**How**：

1. **竞品识别**：
   - 种子搜索：以"品牌名 + 产品类型 + 对比/推荐"为 Query，抓取 AI 平台返回结果中的高频共现品牌
   - 知识库补全：若种子搜索结果不足，调用行业品类图谱补充主流玩家
   - 人工校验：输出候选列表，由用户确认或替换

2. **同步基准测试**：
   - 将 Stage 1 生成的用户 Query 同步应用于每个竞品
   - 记录每个竞品在相同 Query 下的 AI 平台提及率、排名位置、信息完整度

**竞品识别与基准输出示例**：

```json

    },
```

#### Stage 4：AI 搜索场景测试（AI_SEARCH）

**What**：在目标 AI 搜索平台上，使用 Stage 1 生成的用户 Query 进行真实搜索测试，记录品牌在 AI 回答中的**提及率、信息完整度、引用位置、情感倾向**。

**Why**：这是 GEO 诊断的**核心观测阶段**，直接度量"品牌在 AI 原生搜索环境中的存在感"。

**How**：

**搜索测试矩阵设计**：

| 维度 | 配置项 | 说明 |
|------|--------|------|
| **平台选择** | 豆包、ChatGPT、Perplexity、Kimi、文心一言 | 根据用户目标市场确定，可单选或多选 |
| **Query 来源** | Stage 1 生成的 18 条典型 Query | 覆盖 3 类用户 × 6 条意图 |
| **测试模式** | 冷启动（无历史会话）+ 标准提示词 | 消除个性化推荐干扰 |
| **记录字段** | 是否提及、提及位置（首段/中段/尾段）、信息完整度、引用来源、竞品共现 | 结构化记录 |

**关键指标定义**：

| 指标 | 计算方式 | 说明 |
|------|----------|------|
| **提及率** | 提及 Query 数 / 总测试 Query 数 × 100% | 核心指标，反映基础可见度 |
| **首段提及率** | 首段提及 Query 数 / 总提及 Query 数 × 100% | 高价值指标，首段=AI优先推荐 |
| **信息完整度** | 单条回答中品牌信息完整度评分（0-100） | 是否包含产品特点、价格、购买渠道等 |
| **竞品对比位次** | 在同 Query 中，品牌被提及的先后顺序 | 第1位/第2位/未提及 |

---

#### Stage 5：GEO 效果汇总（GEO_EFFECT）

**What**：聚合 Stage 4 的多平台搜索数据，生成跨平台的 GEO 效果总览。包含：品牌整体提及率、平台差异分析、高频共现竞品矩阵、信息缺失模式识别。

**Why**：单一平台数据不足以支撑战略决策。Stage 5 将分散的搜索测试结果转化为**可行动的洞察**。

**How**：
1. **跨平台聚合**：按平台分组计算提及率、完整度，识别平台差异
2. **竞品共现矩阵**：统计品牌与竞品在同一回答中共同出现的频率
3. **信息缺失模式**：分析"未提及"Query 的共性特征，归类缺失场景

---

#### Stage 6：舆情健康度扫描（SENTIMENT）

**What**：抓取黑猫投诉、DoNews、什么值得买、小红书、微博等平台的用户讨论内容，计算品牌的负面率、情感分布、风险等级。

**Why**：AI 搜索引擎在生成回答时，会**主动回避引用负面信息集中或可信度存疑的品牌**。

**How**：

**舆情数据源与权重**：

| 平台 | 数据类型 | 权重 | 抓取方式 |
|------|----------|------|----------|
| 黑猫投诉 | 投诉帖 | 30% | 平台搜索 API |
| 什么值得买 | 评测/开箱 | 20% | 搜索 + 内容解析 |
| 小红书 | 用户笔记 | 20% | 搜索 + 情感分析 |
| 微博 | 公开讨论 | 15% | 搜索 API |
| DoNews/36氪 | 媒体报道 | 15% | 媒体库检索 |

**舆情评分计算**：

| 指标 | 计算方式 | 风险阈值 |
|------|----------|----------|
| **负面率** | 负面内容数 / 总抓取内容数 × 100% | >30% 中风险，>50% 高风险 |
| **情感分布** | 正面:中性:负面 比例 | 正面<20% 触发关注 |
| **风险等级** | 综合负面率 + 投诉严重性 + 传播热度 | 低/中/高三级 |

---

#### Stage 7：综合总览（OVERVIEW）

**What**：基于前 6 个阶段的全部数据，生成诊断报告的"执行摘要"：核心亮点（3-5 条）、核心风险（3-5 条）、总结性判断（一句话定性）。

**Why**：面向品牌决策者的输出需要**结论先行**。Stage 7 将复杂数据压缩为可快速消费的决策支持信息。

**How**：
1. **亮点提取**：从各阶段高分项中筛选具有战略价值的亮点
2. **风险排序**：基于风险等级、影响范围、修复难度三维排序
3. **定性判断**：生成一句话总结，如"品牌具备基础可见度，但基建薄弱和舆情风险制约了 GEO 效果的天花板"

---

#### Stage 8：AIVO 评分（AIVO_SCORE）

**What**：基于前 7 个阶段的数据，计算 AIVO（AI Visibility & Optimization）综合评分。采用 4 维度 × 25% 权重的加权评分模型。

**Why**：AIVO 评分是诊断报告的**核心量化产出**，使品牌能够横向与竞品对比、纵向追踪优化进度、向管理层汇报 GEO 建设 ROI。

**How**：

**评分体系架构**：

| 维度 | 权重 | 代码 | 数据来源 | 计算逻辑 |
|------|------|------|----------|----------|
| **AI 搜索可见度** | 25% | AI_SEARCH_VISIBILITY | Stage 4 搜索测试 | 提及率×40% + 首段提及率×30% + 信息完整度×30% |
| **基建完善度** | 25% | INFRA_COMPLETENESS | Stage 2 基建评估 | 官网健康度×40% + 自媒体矩阵×35% + 权威媒体×25% |
| **竞品对比优势** | 25% | COMPETITIVE_ADVANTAGE | Stage 3 竞品基准 | 与竞品平均分差距的归一化得分 |
| **舆情健康度** | 25% | SENTIMENT_HEALTH | Stage 6 舆情扫描 | (100-负面率)×60% + 正面率×25% + 风险等级修正×15% |

**评分等级标准**：

| 等级 | 分数区间 | 释义 | 典型特征 |
|------|----------|------|----------|
| 优秀 | 90-100 | GEO 领导者 | AI 搜索首段高提及，基建完善，舆情正向 |
| 良好 | 80-89 | GEO 先进者 | 可见度稳定，基建无明显短板，舆情中性偏正 |
| 中等 | 70-79 | GEO 跟随者 | 基础可见度存在，但存在明显提升空间 |
| 较差 | 60-69 | GEO 滞后者 | 可见度低或基建薄弱，竞品优势明显 |
| 差 | 0-59 | GEO 缺失者 | 系统性缺失，AI 平台中几乎无存在感 |

---

#### Stage 9：建议系统（SUGGESTION）

**What**：基于 AIVO 评分结果和各维度诊断数据，生成可执行的优化建议体系。包含：优先级行动清单（P0/P1/P2）、Quick Wins（低投入高回报项）、三阶段路线图（即时/短期/长期）。

**Why**：诊断的价值在于**驱动行动**。建议系统打通"问题识别 → 解决方案 → 预期效果"的闭环。

**How**：

**建议生成规则**：
1. **问题 → 建议映射**：基于 Stage 7 的风险清单，调用"问题-方案"知识库生成对应建议
2. **优先级排序**：综合 `impactLevel`（影响程度）和 `effortLevel`（实施难度）计算优先级得分
3. **得分预测**：基于历史优化案例数据，估算执行建议后的预期得分提升

**建议数据结构（7 属性标准）**：

```json
{
  "id": "SG001",
  "priority": "高",
  "category": "基建升级",
  "dimension": "INFRA_COMPLETENESS",
  "title": "官网补充结构化数据标记（Schema.org/Product）",
  "description": "在官网产品页添加Schema.org Product结构化数据，包含价格、品牌、评分等字段",
  "impactLevel": "高",
  "effortLevel": "quick_win",
  "timeline": "1-2周",
  "expectedImprovement": "+8分",
  "responsibleParty": "技术团队/SEO负责人"
}
```

## 4. 对话交互与数据输出规范

### 4.1 对话交互流程设计

**交互节点总览**：

```
[用户启动] 
    ↓
[节点①] 意图确认 → 确认品牌名称 + 产品类型 + 官网（可选）
    ↓
[节点②] 平台选择 → 确认诊断平台（豆包/ChatGPT/Perplexity 等）
    ↓
[节点③] 阶段1启动确认 → 告知用户即将开始"品牌信息搜索收集"
    ↓
[阶段1执行中] 静默运行（后台并行搜索组A+组B）
    ↓
[节点④] 阶段1完成回传 → 输出：用户画像 + 基建评估 + 竞品分析
    ↓
[节点⑤] 阶段2+3启动确认 → 加载平台参数，提示"正在模拟AI搜索场景"
    ↓
[阶段2+3执行中] 并行搜索数据回传
    ↓
[节点⑥] 阶段2+3完成回传 → 输出：AI收录情况 + 可见性分析 + 舆情扫描
    ↓
[节点⑦] 阶段4评分 → 4维度评分计算
    ↓
[节点⑧] 最终合并 → JSON修复 → HTML报告构建
    ↓
[节点⑨] 交付确认 → 向用户呈现报告摘要 + 文件下载
```

**对话节点详细规范**：

| 节点 | 触发时机 | Agent 输出内容 | 用户输入要求 | 异常处理 |
|------|----------|----------------|--------------|----------|
| ① 意图确认 | 用户首次发送"帮我诊断..."类意图 | 确认品牌名称、产品类型、官网地址 | 补充或确认三要素 | 品牌名称模糊时，提供候选列表让用户选择 |
| ② 平台选择 | 三要素确认后 | 列出支持平台，默认推荐"豆包" | 确认或切换平台 | 不支持平台时提示"暂不支持，建议切换" |
| ③ 阶段1启动 | 平台确认后 | 告知即将开始信息搜索，预计2-3分钟 | 无（自动推进） | 用户追问时，回复"正在后台搜索中，请稍候" |
| ④ 阶段1回传 | 组A+组B搜索完成 | 输出用户画像、基建评估、竞品分析摘要 | 无（自动推进） | 某模块数据缺失时，标记"数据不足"并继续 |
| ⑤ 阶段2+3启动 | 阶段1完成后 | 告知正在执行AI搜索模拟和舆情扫描 | 无（自动推进） | — |
| ⑥ 阶段2+3回传 | 搜索+舆情完成 | 输出AI收录、可见性、舆情分析摘要 | 无（自动推进） | 舆情数据抓取失败时，降级为"基于有限样本" |
| ⑦ 阶段4评分 | 全量数据就绪 | 展示4维度评分计算过程 | 无（自动推进） | 某维度数据不足时，按最低权重折算 |
| ⑧ 最终合并 | 评分完成 | 执行JSON合并与修复，构建HTML | 无（自动推进） | JSON格式错误时，触发修复规则 |
| ⑨ 交付确认 | 报告生成完毕 | 输出AIVO总分、摘要、建议概览、文件路径 | 无（会话结束） | 文件生成失败时，输出纯文本摘要作为兜底 |

### 4.2 输入验证与引导机制

**三要素验证规则**：

| 字段 | 必填 | 验证规则 | 模糊处理策略 | 示例 |
|------|------|----------|--------------|------|
| 品牌名称 | 是 | 非空，长度2-20字符，去除通用后缀 | 若用户输入"格力"，提供候选：珠海格力电器股份有限公司 / 格力品牌 / 其他 | "听力熊" → 直接使用；"格力" → 确认是否指格力电器 |
| 产品类型 | 是 | 非空，需包含品类关键词 | 过于宽泛时，追问细分场景 | "儿童AI对话智能体" ✓；"电子产品" → 追问具体类型 |
| 官网地址 | 否 | 若提供，验证URL格式 | 格式错误时提示修正，无法访问时标记"待验证" | "www.example.com" → 自动补全为 "https://www.example.com" |

| 逗号冗余（末尾逗号） | 扫描 `,` 后紧跟 `}` 或 `]` 的模式 | 删除冗余逗号 |
| 键值缺失 | 验证必填字段存在性 | 缺失字段填充默认值 |

| 字段 | 默认值 | 填充规则 |
|------|--------|----------|

| `sentiment.negativeRate` | -1 | -1表示"未获取到数据" |
| `competitors` | `[]` | 空数组，前端展示"竞品识别失败" |

## 5. AIVO 评分算法与建议系统

### 5.1 AIVO 评分体系总览

**AIVO（AI Visibility & Optimization）评分**是 GEO 可见度诊断师的核心量化评估模型，通过 4 个独立维度对品牌在 AI 搜索生态中的数字存在感进行加权评分，最终输出 0-100 的综合得分及对应的诊断等级。

$$\text{AIVO}_{\text{总分}} = \sum_{i=1}^{4} W_i \times S_i = 0.25 \times S_{\text{可见度}} + 0.25 \times S_{\text{基建}} + 0.25 \times S_{\text{竞品}} + 0.25 \times S_{\text{舆情}}$$

| 维度 | 权重 | 代码标识 | 计算来源 | 说明 |
|------|------|----------|----------|------|
| AI 搜索可见度 | 25% | `AI_SEARCH_VISIBILITY` | Stage 4 + Stage 5 | 品牌在典型用户问题中被 AI 平台提及的比率 |
| 基建完善度 | 25% | `INFRA_COMPLETENESS` | Stage 2 | 官网质量、自媒体矩阵、权威媒体覆盖的完整程度 |
| 竞品对比优势 | 25% | `COMPETITIVE_ADVANTAGE` | Stage 3 + Stage 5 | 与 3-5 家自动识别竞品在相同问题下的相对表现 |
| 舆情健康度 | 25% | `SENTIMENT_HEALTH` | Stage 6 | 负面信息占比、情感分布趋势、风险等级 |

**评分等级标准**：

| 等级 | 分数区间 | 颜色标识 | 业务解读 |
|------|----------|----------|----------|
| 优秀 | 90-100 | 🟢 | 品牌在 AI 搜索生态中具备强可见度，基建完善，舆情健康 |
| 良好 | 80-89 | 🟢 | 整体表现稳健，存在局部优化空间 |
| 中等 | 70-79 | 🟡 | 可见度存在明显短板，需针对性改进 |
| 较差 | 60-69 | 🟠 | 多维度得分偏低，AI 生态存在感弱 |
| 差 | 0-59 | 🔴 | 品牌在 AI 搜索中几乎不可见，需系统性重建 |

### 5.2 四维评分算法详解

#### 维度一：AI 搜索可见度（AI_SEARCH_VISIBILITY）

$$S_{\text{可见度}} = \frac{\sum_{j=1}^{N} M_j \times C_j \times P_j}{N} \times 100$$

其中：
- $N$ = 典型问题总数
- $M_j$ = 问题 $j$ 中品牌是否被提及（0 或 1）
- $C_j$ = 语境权重（正面=1.0，中立=0.7，负面=0.3，未提及=0）
- $P_j$ = 位置权重（首屏=1.0，折叠区=0.6，未提及=0）

**示例计算**（听力熊，10 个问题）：

| 问题 | 提及 | 位置 | 语境 | 加权得分 |
|------|------|------|------|----------|
| "儿童AI学习机推荐" | 是 | 首屏 | 中立 | 0.7 |
| "听力熊和小度哪个好" | 是 | 首屏 | 负面 | 0.3 |
| "小度学习机怎么样" | 否 | - | - | 0.0 |
| ... | ... | ... | ... | ... |
| **合计** | - | - | - | **5.8 / 10** |

$S_{\text{可见度}} = 5.8 / 10 \times 100 = 58$ 分

#### 维度二：基建完善度（INFRA_COMPLETENESS）

$$S_{\text{基建}} = 0.4 \times S_{\text{官网}} + 0.3 \times S_{\text{自媒体}} + 0.3 \times S_{\text{权威媒体}}$$

**第一层：官网质量（40%）**

| 检查项 | 权重 | 检测方式 | 得分逻辑 |
|--------|------|----------|----------|
| 官网可访问性 | 20% | HTTP 状态码检测 | 200=满分，超时/403=0分 |
| 结构化数据标记 | 25% | 检测 Schema.org / JSON-LD | 完整标记=1.0，部分=0.5，无=0 |
| 核心页面完整度 | 25% | 检测 About / Product / Contact 页面 | 三页齐全=1.0，缺页按比例扣分 |
| 内容更新频率 | 15% | 检测最新内容发布日期 | 30 天内有更新=1.0，90 天内=0.5，否则 0 |
| 移动端适配 | 15% | 响应式检测 | 完美适配=1.0，可用但体验差=0.5，不可用=0 |

**第二层：自媒体矩阵（30%）**

| 平台 | 检查项 | 存在即得分 | 活跃加分 |
|------|--------|------------|----------|
| 微信公众号 | 账号存在 + 近期发文 | 0.5 | 30 天内有更新 +0.5 |
| 小红书 | 品牌号/企业号 | 0.5 | 30 天内有笔记 +0.5 |
| 知乎 | 机构号/品牌话题 | 0.5 | 30 天内有内容 +0.5 |
| 今日头条 | 企业号 | 0.5 | 30 天内有更新 +0.5 |
| B站 | 官方账号 | 0.5 | 30 天内有视频 +0.5 |

**第三层：权威媒体覆盖（30%）**

| 媒体类型 | 检测方式 | 权重 |
|----------|----------|------|
| 行业垂直媒体（36氪、极客公园等） | 搜索品牌名 + site 限定 | 40% |
| 主流科技媒体（虎嗅、DoNews等） | 同上 | 30% |
| 权威财经/综合媒体 | 同上 | 30% |

#### 维度三：竞品对比优势（COMPETITIVE_ADVANTAGE）

$$S_{\text{竞品}} = \frac{S_{\text{品牌}}}{\frac{1}{N}\sum_{k=1}^{N} S_{\text{竞品}k}} \times 100$$

**示例**（听力熊 vs 竞品）：

| 品牌 | AI 搜索可见度得分 | 相对比值 |
|------|-------------------|----------|
| 小度 | 85 | 基准 |
| 汤姆猫 | 82 | - |
| 阿尔法蛋 | 78 | - |
| 小米 | 75 | - |
| 优必选 | 65 | - |
| **听力熊** | **58** | **58/77 = 75.3 分** |

#### 维度四：舆情健康度（SENTIMENT_HEALTH）

**Step 1：多源舆情抓取**

| 数据源 | 抓取内容 | 权重 |
|--------|----------|------|
| 黑猫投诉 | 投诉数量、投诉类型、解决率 | 30% |
| 消费评价平台（什么值得买等） | 用户评价、评分 | 25% |
| 社交媒体（微博、小红书） | 用户讨论、情感倾向 | 25% |
| 新闻媒体 | 负面报道、公关危机 | 20% |

**Step 2：风险等级判定**

| 负面率 | 风险等级 | 得分区间 |
|--------|----------|----------|
| < 10% | 低风险 | 85-100 |
| 10%-20% | 中低风险 | 70-84 |
| 20%-30% | 中风险 | 55-69 |
| 30%-50% | 高风险 | 35-54 |
| > 50% | 极高风险 | 0-34 |

**Step 3：最终舆情得分**

$$S_{\text{舆情}} = \max(0, 100 - \text{负面率} \times 2 - \text{风险等级扣分})$$

### 5.3 建议系统（SUGGESTION）

#### 建议生成规则引擎

每条建议由规则引擎根据以下触发条件生成：

| 触发条件 | 生成建议类型 | 示例 |
|----------|-------------|------|
| 某维度得分 < 70 | 该维度系统性改进建议 | "基建完善度仅 55 分，建议优先补齐自媒体矩阵" |
| 某检查项 = 0 分 | 具体修复项 | "官网缺少结构化数据标记，建议添加 JSON-LD" |
| 负面率 > 20% | 舆情管理建议 | "负面率 30%，建议建立投诉响应机制" |
| 竞品在某维度显著领先 | 对标赶超建议 | "竞品小度在权威媒体覆盖上领先 40%，建议加强 PR 投入" |

#### 建议优先级判定矩阵

| 优先级 | 判定条件 | 最大数量 |
|--------|----------|----------|
| 高 | impactLevel=高 AND effortLevel≠heavy | 最多3条 |
| 中 | impactLevel=中 OR (impactLevel=高 AND effortLevel=heavy) | 最多4条 |
| 低 | impactLevel=低 OR effortLevel=heavy | 最多3条 |

#### 建议条目完整属性

```json
{
  "id": "SUGG-001",

  "priority": "高",

```

#### 三阶段路线图

| 阶段 | 时间范围 | 目标 | 典型内容 |
|------|----------|------|----------|
| **P1 即时** | 0-4 周 | 止血 + 速赢 | Quick Wins、0 分项修复、紧急舆情应对 |
| **P2 短期** | 1-3 月 | 系统性提升 | 基建补齐、内容矩阵搭建、媒体关系建立 |
| **P3 长期** | 3-6 月 | 竞争力构建 | 品牌权威建设、用户口碑运营、GEO 策略深化 |

#### 得分预测模型

```json
{
  "scoreProjection": {
    "current": 57,
    "afterP1": 65,
    "afterP2": 75,
    "afterP3": 82,
    "breakdown": {
      "AI_SEARCH_VISIBILITY": { "current": 58, "projected": 72 },
      "INFRA_COMPLETENESS": { "current": 55, "projected": 80 },
      "COMPETITIVE_ADVANTAGE": { "current": 60, "projected": 68 },
      "SENTIMENT_HEALTH": { "current": 56, "projected": 70 }
    }
  }
}
```

**预测逻辑**：
- 每条建议的 `expectedImprovement` 按维度累加
- 同一维度的多条建议取最高提升值（避免重复计算）
- 总分预测受限于 100 分上限和行业基准天花板

### 5.4 算法迭代与校准机制

| 校准项 | 方法 | 频率 |
|--------|------|------|
| 权重校准 | 分析执行建议后的实际得分变化 vs 预测差异，调整维度权重 | 每季度 |
| 竞品基准 | 更新行业竞品库，重新计算行业均值线 | 每月 |
| 检查项清单 | 根据 AI 平台算法变化（如豆包更新），调整基建检查项 | 每季度 |
| 阈值校准 | 根据累积诊断数据，调整评分等级阈值 | 每半年 |
